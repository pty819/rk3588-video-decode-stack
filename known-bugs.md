# Known bugs

> Bugs we know about, that we have not fixed. Documented so
> future maintainers don't re-discover them.

## HEVC ST RPS — Collabora detlev/ffmpeg `wip: fix st rps`

**Severity:** medium (specific HEVC content fails)
**Affected:** ffmpeg 8.0.1 with HEVC BluRay rips and certain
HEVC Main 10 content
**Workaround:** see below

### The bug

The Collabora detlev/ffmpeg fork's HEAD is
`dfa10f6 wip: fix st rps` (2026-01-15). The commit
description:

```text
wip: fix st rps
```

The `wip:` prefix is a self-imposed "not done" marker. The
commit addresses the short-term reference picture set (ST RPS)
code path in `libavcodec/v4l2_request_hevc.c`, but the fix is
incomplete. The `hevc_ext_sps_*_rps` UAPI is correctly written
to the kernel, but the RPS validation in userspace is missing
edge cases for:

- B-frames with reference picture list reordering
- Long-term reference frames with broken entries
- Stream features where the ST RPS has more entries than
  the SPS-declared maximum

When you hit one of these, the kernel may:

- Silently drop the frame (most common)
- Return `EINVAL` from `VIDIOC_S_EXT_CTRLS` (you'll see
  `failed to set hevc_ext_sps_*_rps: -EINVAL` in the log)
- Worse, return success but with garbage data in the RPS,
  leading to a corrupted decoded frame

### What content triggers it

- HEVC Main profile 8-bit, simple GOP: **does not** trigger
- HEVC Main 10 (10-bit color): **sometimes** triggers, depends
  on the encoder
- HEVC BluRay rips with B-frames and complex RPS: **often** triggers
- HEVC 4K from a camcorder (simple IP frames): **does not** trigger

### What content doesn't

In our testing:

- 4K HEVC test patterns generated by libx265 ultrafast: ✓
- 4K HEVC Main 8-bit from a Sony A7S camera: ✓
- 4K HEVC Main 10 from a Blackmagic camera: ✓
- 1080p HEVC from YouTube downloads: ✓
- 4K HEVC BluRay rip with B-frames: ⚠️ intermittent

### Workarounds

1. **gstreamer.** Use `v4l2slh265dec` instead of ffmpeg's
   `v4l2request` hwaccel. GStreamer's RPS handling is
   independent of ffmpeg's. We've seen it work on the same
   BluRay rips that fail in ffmpeg.

   ```bash
   gst-launch-1.0 -v filesrc location=bluray-hevc.mkv ! \
     matroskademux ! h265parse ! v4l2slh265dec ! \
     videoconvert ! waylandsink
   ```

2. **Switch to LibreELEC jernejsk ffmpeg 7.1.** This is
   the production stack used by Kodi. Loses the
   ext-SPS-RPS improvements in 8.0.1 but has a stable
   HEVC RPS implementation.

3. **Wait for Collabora.** The `wip: fix st rps` commit
   has been stale since 2026-01-15. Either Collabora is
   waiting on a kernel-side UAPI change, or Detlev is
   busy. No public ETA.

### What we do

For our daily use, we use the ffmpeg 8.0.1 fork for all
content except BluRay rips. For BluRay rips, we use
gstreamer. The verify scripts
(see [`verify`](verify)) test with synthetic content
that doesn't trigger the bug; this is intentional.

## `RPS validation` warnings in dmesg

**Severity:** low (cosmetic)
**Affected:** all HEVC content with non-trivial RPS

You'll see this in dmesg after a 4K HEVC decode:

```text
[12345.678] rockchip-vdec2 vdec2: hevc: rps validation failed for frame 42
```

This is a debug message, not an error. The kernel still
decodes the frame correctly. The message is from
`drivers/staging/media/rockchip/vdec/vdec.h` and is not
related to the Collabora fork's `wip: fix st rps` commit;
they're two different RPS code paths in two different
places.

## CmaTotal too small for AV1 4K

**Severity:** medium (specific codec)
**Affected:** 4K AV1 content
**Workaround:** CMA 1G

The Collabora fork has `av1_v4l2request` enabled. We've
seen it decode 4K AV1 test patterns correctly, but
reference frame allocation can fail with the default
512M CMA. The rkvdec's AV1 path uses significantly more
CMA than H.264 / HEVC due to the superblock structure.

To use AV1 4K reliably:

```text
# /boot/armbianEnv.txt
extraargs=cma=1024M
```

This conflicts with the Mali-G610 GPU's memory pool at
higher resolutions. We don't recommend it for general
use; only for AV1 4K.

## `libavfilter/vf_libplacebo.c:940` was renamed in newer ffmpeg

**Severity:** cosmetic (already fixed)
**Status:** already patched

The fork's `libavfilter/vf_libplacebo.c:940` references
`PL_ALPHA_NONE`, which was renamed in libplacebo 7.x to
`PL_ALPHA_UNKNOWN`. The fork's HEAD `dfa10f6` (2026-01-15)
still uses the old name in this one location.

**Workaround:** the one-line patch documented in
[`install`](install). The patch is required; without it
the build fails.

**Long-term:** when Collabora re-bases against a newer
ffmpeg that has the rename, this will fix itself. Track
the `wip: fix st rps` commit for the next rebase.

## `apt build-dep mpv` doesn't work

**Severity:** low (workaround exists)
**Status:** worked around

`apt build-dep mpv` requires `deb-src` URIs in
`/etc/apt/sources.list`, which armbian noble doesn't
include by default.

**Workaround:** install the mpv build deps manually.
The list is in [`install`](install) and in
[`mpv-rejected`](mpv-rejected).

**Long-term:** this is an armbian packaging decision,
not a stack bug. If you want to fix it, file an issue
against the armbian noble pocket config.

## What we don't have bugs in

- `v4l2request` hwaccel itself — stable in the Collabora
  fork for 5+ codecs (H.264, HEVC, VP8, VP9, AV1, MPEG-2)
- The build system — no known build issues beyond the
  PL_ALPHA_NONE patch
- The install process — no known install issues
  beyond the apt-mark hold requirement
- The coexistence with apt ffmpeg 6.1.1 — SONAMEs
  prevent any conflict
- The verify scripts — all 5 checks pass on a clean
  build
